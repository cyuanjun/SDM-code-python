## Resolved

- ~~**No RBAC / menu gating in `app.py`.**~~ **Resolved 2026-05-15.** Sidebar now filters pages by the logged-in user's role (looked up via `ViewUserProfileController.view_user_profile`). Not signed in → only `Log In` + `.info (debug)`. Each role only sees its own actor's pages. `.info (debug)` stays visible for development; remove or hide along with the debug page before final demo.
- ~~**Email is not unique on `UserAccount`.**~~ **Resolved 2026-05-15** (lecturer instruction). `user_account.email` now has a `UNIQUE` constraint at the schema level. `UserAccount.create_account` returns `Optional[UserAccount]` — `None` on conflict; `UserAccount.update_user_account` returns `False` on conflict. Boundary surfaces "email already in use" errors. US-6 diagram still types the signature as `createAccount(...) -> UserAccount` with no failure branch — that diagram fix is now part of the existing "create / login failure branch" entry in [diagram_typos.md](diagram_typos.md).
- ~~**Role is not unique on `UserProfile`.**~~ **Resolved 2026-05-17.** `user_profile.role` now has a `UNIQUE` constraint at the schema level. `UserProfile.create_profile` returns `Optional[UserProfile]` — `None` on conflict; `UserProfile.update_user_profile` returns `False` on conflict (also covers the case where an update would collide with another row's role). Boundary surfaces "A profile with that role already exists." The role is the join key for `PAGES_BY_ROLE` in `app.py`, so allowing duplicates was a footgun — one profile per role matches the natural model (one role definition, many accounts FK into it). Diagram still types `role: String`; the UNIQUE constraint is a schema-level guard not modelled on the class diagram (same shape as the email-unique decision, no diagram update needed).
- ~~**Category name is not unique on `FundraisingActivityCategory`.**~~ **Resolved 2026-05-17.** `fundraising_activity_category.category_name` now has a `UNIQUE` constraint at the schema level. `FundraisingActivityCategory.create_category` returns `Optional[...]` — `None` on conflict; `update_fundraising_activity_category` returns `False` on conflict. Boundary surfaces "A category with that name already exists." Parallel to role uniqueness — one category per name matches the curated-list intent.
- ~~**`category` on `FundraisingActivity` is unlinked from `FundraisingActivityCategory`.**~~ **Resolved 2026-05-18.** The 2026-05-18 US-13 diagram replaces `category: String` with `FRACatId: String` (FK to `FundraisingActivityCategory`), cascading the attribute change through US-14/15/17/20/21/30/31. `fundraising_activity.fra_cat_id` is a `TEXT NOT NULL REFERENCES fundraising_activity_category(fra_cat_id)` FK at the schema level. The create / update boundaries are now `st.selectbox` populated from `FundraisingActivityCategory.view_all_categories()` (active categories only); fundraiser can no longer type a free-text category and bypass the curated list. Search methods (US-17, US-20, US-30, donation history, favourite search) JOIN to `fundraising_activity_category` and match against `category_name` so existing test fixtures and UX continue to work.
- ~~**`updateMyFundraisingActivity` took a `FundraisingActivity` DTO instead of unpacked fields.**~~ **Resolved 2026-05-18.** The US-15 diagram defines six editable fields (`title, description, target_amount, FRACatId, start_date, end_date`); code was wrapping them into a `FundraisingActivity` object and passing it through. Refactored entity + controller to take the six fields unpacked, matching the diagram contract exactly. As part of the same sweep, `completed` and `suspended` are no longer settable through update — `completed` is now derived from `end_date < today` (the `completed` column was dropped from `fundraising_activity` and replaced with a Python `@property` on the entity), and `suspended` was always owned by US-16 / Exception A unsuspend anyway. The diagram still types `completed: Boolean` on the entity — that's honoured because the attribute still exists on instances; only the storage is implementation detail. A future re-export could mark it `/completed: Boolean` (derived) for clarity, but it's not a hard deviation. US-30 (`search_my_completed_fundraising_activity`) and US-31 (`view_my_completed_fundraising_activities`) read methods now filter `WHERE end_date < ?` instead of `WHERE completed = 1`.
- ~~**Suspended fundraising activities still surfaced to donees via Browse / Search / Favourites.**~~ **Resolved 2026-05-18** (business rule). The reworked US-16 puts suspend at the fundraiser-owner's discretion, but the donee-facing reads were oblivious to the `suspended` flag — a donee could still find a suspended activity through `[Browse Fundraising Activities]`, search results, or `[My Favourites]`. After this fix, donee-facing entity reads filter `WHERE suspended = 0`: `view_all_fundraising_activities`, `search_fundraising_activity` (US-20), `Favourite.view_favourite_list` (US-24), `Favourite.search_favourite` (US-25). Owner-scoped reads (`view_my_fundraising_activities`, `view_my_fundraising_activity`, `search_my_fundraising_activity`, `search_my_completed_fundraising_activity`, `view_my_completed_fundraising_activities`) are unchanged — the fundraiser still sees their own suspended activities so they can edit / unsuspend. Donation history (US-32/33) is unchanged — historical record, suspended doesn't retroactively hide donations. Favourite rows themselves are not deleted on suspend — they reappear automatically when the owner unsuspends. Diagram still types `viewMyFundraisingActivity` / `searchFundraisingActivity` without a `suspended` parameter; the filter is a schema-implicit business rule the same shape as `WHERE owner_account_id = ?` for ownership.
